Axiomas Fundamentais da Lógica

Princípio da não contradição: uma proposição não pode ser verdadeira e falsa ao mesmo tempo.

Princípio do terceiro excluído: toda proposição ou é verdadeira ou é falsa, isto é, verifica-se sempre um desses dois casos, mas nunca um terceiro.

Proposições Compostas

As proposições atômicas, ou simples, são aquelas contidas em si mesmo. Que representaremos por letras $p, q, r, s$.

Proposições compostas são aquelas representadas por uma fórmula lógica. São proposições criadas por proposições atômicas e Oper., ou concetivos.

Oper. Lógicos - AND $(\land)$

Paula compra tijolos e faz casas. é uma proposição composta

$p = $ Paula compra Tijolos

$q = $ Paula faz casas

Será representada por: $p \land q$

Oper. Lógicos - AND $(\land)$

Estudei muito, mas não fui aprovado.

$p = $ Eu estudei muito.

$q = $ Eu não fui aprovado.

Será representada por: $p \land q$

Oper. Lógicos - AND $(\land)$

Estudei muito, contudo não fui aprovado.

$p = $ Eu estudei muito.

$q = $ Eu não fui aprovado.

Será representada por: $p \land q$

Oper. Lógicos - OR $(\lor)$

Sílvia vai ao cinema ou Jéssica vai a Praia.

$p = $ Sílvia vai ao cinema.

$q = $ Jéssica vai a Praia.

Será representada por: $p \lor q$

Oper. Lógicos - negação $(\neg)$

O carro é de qualquer cor menos preto.

Será representada por: $p = \neg p$ o carro não é preto.

Oper. Lógicos - implicação $(\Rightarrow)$

Se Janaína é arquiteta então Bruna é médica

$p = $ Janaína é arquiteta.

$q = $ Bruna é médica.

Será representada por: $p \Rightarrow q$

Oper. Lógicos - implicação $(\Rightarrow)$

No caso da implicação a proposição $p$ é a hipótese, ou antecendente, e a proposição $q$ a conclusão ou consequente.

A operação condicional, ou implicação, indica que $p$ é uma condição para $q$, sem $p$, não há $q$.

$p \Rightarrow q$ é lido com p implica em q

Oper. Lógicos - implicação $(\Rightarrow)$

$p \Leftarrow q$

  • quando $p$ então $q$;
  • se $p$, então $q$;
  • $p$ é suficiente para $q$;
  • $q$ se $p$;
  • uma condição nessária para $p$ é $q$;
  • $p$ sempre que $q$;
  • $q$ é necessário para $p$.

Oper. Lógicos - bi-implicação $(\Leftrightarrow)$

Será aprovado se, e somente se, estudar.

$p = $ Será aprovado.

$q = $ Se estudar.

Será representada por: $p \Leftrightarrow q$

Se tirar 10 na prova, lhe darei R$1000,00

  • A proposição será verdadeira se eu cumprir a promessa. Caso contrário será falsa.
  • Suponha que você tirou 10 na prova, então $p = T$ e eu mantenho a promessa, logo $q = T$. A proposição é verdadeira.
  • Suponha que você tirou 10 na prova, então $p = T$ e eu não cumpro a promessa, logo $q = F$. A proposição é falsa.
  • Suponha que você tirou 8 na prova, se eu lhe dou os R$ 1000,00 não importa por que eu não tive a chance de quebrar a promessa. A proposição é verdadeira.

Repositório

Localização preferencialmente segura para armazenar todos os artefatos de projeto.

Edite seu arquivo localmente em uma working tree depois coloque seus arquivos em repositório usando o comando commit.

No GIT, cada usuário pode manter sua própria cópia do repositório.

Repositório

Você precisa adicionar os arquivos que estão na sua working tree ao repositório.

Não deve incluir arquivos .obj e .exe. Ou outros que sejam derivados do código.

Deve incluir arquivos Make ou de configuração.

Repositório

Podemos criar repositórios em qualquer lugar. Até no computador de desenvolvimento.

Coloque em computadores com RAID e que estejam disponívies 24h/7.

Considere: Github; GitLab; BitBucket

Além da Amazon, Microsoft e Google. Todos têm serviços de CI/CD.

Estrutura de Repositórios

Repositório Local

Working tree ou Directory tree. os arquivos não são controlados pelo GIT. podemos alterar qualquer coisa sem medo.

Staging Area estes são os arquivos que serão incluídos no seu próximo commit. Estes são os arquivos que o GIT conhece.

Repository estes são os arquivos que você já comitou. Está no diretório .git.

Staging Area

Contém uma cópia do seu último commit. Uma lista de arquivos e o hash SHA1 dos arquivos que fizeram parte do seu último commit.

Este índex é, na verdade, um arquivo binário no diretório .git.

Sempre que você altera, renomeia, ou apaga arquivos o SHA1 deste arquivo muda. 

              
                $ git ls-files -s

Fluxo de Trabalho

Crie e modifique tudo que quiser no seu Working Directory.

Stage seus arquivos para que o GIT tenha conhecimento deles.

Commit do seus arquivos para que o GIT faça uma cópia destes arquivos diretamente no seu .git.

GIT na Linha de Comando

Para buscar ajuda: 

              
                $ git help verbo
                $ git verbo --help 
                $ man git-verbo

Existe um Livro gratuito online que está em processo de tradução para o português.

Configurando seu ambiente

              
                $ git config --global user.name “Seu Nome”
                $ git config --global user.email seu@email.com
                $ git config –-list

Estes valores serão ajustados globalmente para todos os seus projetos no GIT

Criando uma cópia local

              
                $ git clone <url> [nome do dir local]
                $ exemplo:
                $ git clone https://github.com/frankalcantara/learngo

Irá criar, no diretório onde você rodar este comando, um diretório chamado learngo com um arquivo .git contendo o histórico do repositório do GitHub.

Para criar um repositório

              
                $ git init

Irá criar, um repositório, no diretório corrente com o respectivo .git. Uma vez que o diretório está criado você pode adicionar arquivos a Stage Area. 

              
                $ git add teste1.c
                $ git commit –m “primeiro commit neste repositorio para testes”

O commando commit incluiu os arquivos que estavam na Stage Area no repositório local.

Comandos

Comando Descrição
git clone url [dir] Faz uma cópia de um repositório em um diretório local
git add files Adiciona arquivos ao controle de versão
git commit Grava uma imagem da Staging Area
git status Verifica o status dos seus arquivos na Working Area e na Stage Area
git pull Recupera arquivos de um repositório e tenta o mergeno seu branch corrente.
git push Coloca suas modificações em um repositório remoto

Até agora trabalhamos localmente

Para ver o status dos seus arquivos no Working Directory e Stage Area. 

              
                $ git status –s

Sem o parâmetro "-s" teremos uma versão mais detalhada do status.

Até agora trabalhamos localmente

Podemos comparar o Working Directory e Stage Area. 

              
                $ git diff

Ou ainda, comparar a Stage Area com o repositório local. 

              
                $ git diff --cached

Até agora trabalhamos localmente

Outra ferramenta importante disponível é o histórico de modificações log. 

              
                $ git log --oneline

Podemos ver também um número reduzdido de modificações. As dez últimas, por exemplo: 

              
                $ git log -10

Boas Práticas

  1. use add e commit no diretório local.
  2. use pull para trazer as modificações recentes do repositório remoto.
  3. use push para levar suas modificações para o repositório remoto.

Trabalhando com diretórios remotos

Para baixar as modificações mais recentes em um repositório remoto no seu repositório local. Além de disponibilizar os arquivos modificados no seu Working Directory. 

              
                $ git pull origin master

Trabalhando com diretórios remotos

Para enviar suas modificações, depois de comitadas, para o repositório remoto. 

              
                $ git push origin master

Branches

Branches, em português, ramos, são derivações da Gráfico Direcional Acíclico que o GIT usa. Usando os branches criamos uma versão do repositório, local ou remoto, para modificações. Do ponto de vista do GIT, tudo que é criado é um ponteiro para este novo ramo. 

            
              $ git branch frank

Branches

A simples criação de um branche, não altera nada no seu trabalho, você irá continuar fazendo commit no Master.

O branch Master não tem nada de especial, e é tão comum apenas por que é criado por padrão quando criamos um repositório. 

            
              $ git log --oneline --decorate

Branches

Para apontar seu desenvolvimento para o novo branch:

              
                $ git checkout frank

Agora tudo que você fizer será feito no branch frank. Mas você pode voltar ao branch Master. 

              
                $ git checkout master

Branches

A ação de colocar todo o trabalho desenvolvido em um branch no master, sem prejudicar o trabalho que pode ter sido feito no master por outros desenvolvedores é chamada de merge. 

                
                  $ git checkout master
                  $ git merge frank

Vários desenvedores, vários branches um produto único.

Branches

Um merge pode criar um conjunto de conflitos quando dois desenvolvedores alterarem as mesmas linhas. O GIT fornece uma lista de conflitos. 

                  
                    $ git status

Existem ferramentas para ajudar na correção de conflitos.
O merge só vai ser executado depois que todos os conflitos forem resolvidos.
